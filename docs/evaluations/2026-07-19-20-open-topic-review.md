# 2026-07-19—20 开放主题复盘与“按主题看海南”方案

## 结论

当前问题不是再给 `config/topics.json` 增加几个枚举值就能解决的。现有字段把“文章具体在谈什么”“所属宽泛领域”“文章体裁”压成同一层受控标签，导致模型只能在十个大筐里多选兜底。最终结果是：标签表面覆盖率很高，实际区分度很低；`文旅体育` 尤其成为文化、历史、文学、非遗、摄影、影视、地方记忆乃至传统农业知识的共同容器。

建议把主题生产拆成两层：

1. **入站时开放识别**：模型直接从全文给出一个自由的核心主题和少量次级主题，不看候选词表。
2. **入站后规范归一**：独立的主题解析阶段把自由主题合并到稳定主题树，必要时提出新叶子节点；导航页使用稳定 ID，文章详情优先展示准确的叶子主题。

这能同时满足两件看似冲突的事：模型不受武断词表限制，未来的“按主题看海南”仍然拥有稳定 URL、可聚合的长期目录和可维护的主题边界。

### 现有标签的四类失真

1. **大类代替主题**：`产业项目` 不能区分沉香林下经济、糟粕醋食品工业、桑蚕产业、医疗 AI、茶业标准化。
2. **多选掩盖不确定性**：40% 的文章被打上三个及以上大类，看似全面，实际没有主次，无法决定应该进入哪个主题页。
3. **关键词触发噪声**：出现“国际”就容易得到 `对外交流`，出现数字就容易得到 `经济数据`，出现自然景物就容易得到 `生态环境`。
4. **体裁与议题混淆**：书评、人物、散文、摄影教程不是主题；它们应是 `content_form`，同时仍需标注作品真正讨论的议题。

最典型的错误是《古法晒盐忙》。全文中心是老盐工以千年技艺晒制海盐，准确主题应为“古法制盐”，可归一到“文化与历史 / 非物质文化遗产 / 古法制盐”，次级主题是“海盐文化”。原结果只有 `文旅体育`，会让“按主题看海南”把非遗生产现场误做成泛旅游内容。

| 走进“三国”更深处 | 文旅体育 | 动画电影评论 | 三国历史叙事、AI 动画 | 影视评论被压成文旅大类 |

## 新的主题模型

### 1. 第一次入库：开放主题，不做目录裁决

第一次语义增强只回答“这篇文章具体在谈什么”。建议用以下结构替代 `topic_mentions`：

```json
{
  "topic_profile": {
    "primary": {
      "name": "古法制盐",
      "evidence": "熟练运用千年古法技艺晒制海盐"
    },
    "secondary": [
      {
        "name": "非物质文化遗产",
        "evidence": "原汁原味的非遗制盐场景"
      },
      {
        "name": "海盐文化",
        "evidence": "海盐文化魅力"
      }
    ]
  },
  "content_form": "photo-news"
}
```

约束建议：

- `primary` 必须且只能有一个；正文为空时可为 `null`。
- `secondary` 为 0—3 个，不允许用上位词把数组填满。
- 名称通常为 2—10 个汉字，应是可被读者理解、可跨文章复用的名词短语。
- 不输出“海南新闻、综合、其他、发展、工作”等低信息词。
- 不因正文出现一个例子、数字、地点或形容词就增加主题。
- 证据继续要求逐字引用原文，保留当前验证优势。
- `topic_candidates` 从模型输入中移除，`config/topics.json` 不再决定模型能看见什么。

### 2. 把内容形式独立出来

建议新增受控的 `content_form`，例如：

`news`、`photo-news`、`feature`、`profile`、`analysis`、`commentary`、`essay`、`explainer`、`interview`、`book-review`、`book-list`、`film-review`。

它只回答“文章怎么写”，不参与主题页语义。例如：

- 《古法晒盐忙》：`photo-news` + “古法制盐”
- 《一座小城》：`essay` + “临高地方记忆”
- 《渗透文明的历史容器》：`book-review` + “河流与城市文明”
- 《海南竞技体育再“加速”》：`analysis` + “海南竞技体育”

### 3. 第二阶段：主题归一与主题树

新增一个不改写原始语义结果的 `topic_resolution` 阶段。它读取自由主题、标题、摘要和证据，执行：

1. 映射到已有规范主题；
2. 合并同义词，如“古法晒盐 / 传统制盐 / 古法制盐”；
3. 选择父级路径；
4. 无合适节点时提出新叶子主题，不允许硬塞最近的大类；
5. 为低置信或新节点进入人工审核队列。

主题目录中的节点建议至少包含：

```json
{
  "topic_id": "traditional-saltmaking",
  "name": "古法制盐",
  "aliases": ["古法晒盐", "传统制盐"],
  "parent_id": "intangible-heritage",
  "definition": "以传统工艺采卤、晒制或加工海盐的生产技艺与传承",
  "include": ["盐田生产现场", "制盐技艺传承"],
  "exclude": ["普通盐业价格", "泛滨海旅游"],
  "status": "active"
}
```

`definition/include/exclude` 比现在只有 `aliases` 更重要，它能让模型和编辑判断主题边界，而不是靠词面碰撞。

### 4. 稳定导航根节点

导航根节点仍应受控，但只作为第二阶段的目录骨架，不作为第一次入库的候选答案。建议初版设为：

1. 自贸港与开放
2. 经济与市场
3. 产业与项目
4. 科技与创新
5. 农业与乡村
6. 旅游与消费
7. 交通与城市
8. 教育与青年
9. 医疗与健康
10. 生态与气候
11. 文化与历史
12. 体育与赛事
13. 社会与公共服务
14. 治理与法治
15. 应急与安全

这些不是展示给每篇文章的最终主题，而是叶子主题的上层入口。例如：

```text
文化与历史
└── 非物质文化遗产
    ├── 古法制盐
    ├── 黎陶
    └── 儋州调声

农业与乡村
├── 热带特色农业
│   ├── 沉香林下经济
│   ├── 海南桑蚕
│   └── 白沙茶
└── 农业知识
    ├── 海南农时
    └── 桑蚕科技推广
```

### 5. “按主题看海南”的收录边界

主题识别和产品收录必须分开：

- 所有文章都可有内容主题，便于站内搜索。
- “按主题看海南”默认只收录 `scope=hainan`，以及确有海南主体参与的 `mixed`。
- `national`、`domestic`、`foreign` 文章即使主题准确，也不自动进入海南主题流。
- 主题页按“核心相关”与“延伸相关”分层：核心主题来自 `primary`，次级主题只进入延伸区。
- 父级页面通过主题树祖先关系自动聚合，不要求模型给每篇文章重复打父标签。

这样，《全球粮食安全面临三大“灰犀牛”》可以被准确标成“全球粮食安全”，但不会因为出现农业或气候就混入“按主题看海南”的农业页。

## 数据与流水线改造建议

### 契约版本

建议新建 `prompts/article-enrichment/v3/`，不要原地改变 v2 历史合同。

- 模型输入移除 `topic_candidates`。
- 模型输出用 `topic_profile` 和 `content_form` 替代 `topic_mentions`。
- 第一次入库仍由交互式 coding agent 完成，不引入外部模型 API。
- v2 历史结果保持只读，迁移时保留到 `legacy_topics` 供审计。

### 公共内容建议字段

```json
{
  "topic_profile": {
    "primary": {"name": "古法制盐", "evidence": "..."},
    "secondary": []
  },
  "resolved_topics": [
    {
      "topic_id": "traditional-saltmaking",
      "name": "古法制盐",
      "relation": "primary",
      "path": ["文化与历史", "非物质文化遗产", "古法制盐"]
    }
  ],
  "content_form": "photo-news",
  "legacy_topics": [
    {"topic_id": "culture-tourism-sports", "name": "文旅体育"}
  ]
}
```

### 新索引

建议新增：

- `content/topics/catalog.json`：规范主题节点与边界定义。
- `content/topics/aliases.json`：自由名称到规范节点的映射。
- `content/indexes/topics.json`：主题首页统计与最近更新。
- `content/indexes/topic-feed/{topic_id}.json`：按主主题和次主题分区的文章流。
- `data/production-json/topic-resolution/`：归一输入、输出与审核记录。

不要在前端从展示名称临时生成 ID；主题 ID 必须由归一阶段稳定分配，改名不改 URL。

## 实施顺序

1. **先定义 v3 合同和验证器**：开放主题、单一主主题、少量次主题、独立体裁。
2. **双写两期样本**：保留 v2 `topics`，同时生成 v3 结果，验证页面与搜索行为。
3. **建立首版主题树**：以本次 70 篇出现的真实叶子主题为种子，不凭空一次性设计完整百科分类。
4. **实现归一审核**：已有节点自动映射；新叶子、歧义映射进入审核。
5. **建设主题索引和页面**：先上线 5—8 个内容量足够的海南主题页，再逐步扩展。
6. **历史回填**：确认合同稳定后，再对 `content/` 中既有文章回填；旧标签只作审计，不作为新主题页来源。

## 验收标准

- 正文存在的文章，核心主题覆盖率不低于 98%。
- 每篇只有一个核心主题；次级主题不超过 3 个。
- 随机抽样中，编辑对核心主题“准确表达正文中心”的认同率不低于 90%。
- 同一持续议题的不同报道能落到同一规范 ID，例如两天的彭水山体崩塌报道。
- 主题页前 20 篇中，核心相关内容占比不低于 85%。
- 新主题允许出现；“新叶子率为零”反而应视为系统重新被封闭词表绑死的预警。
- 文章详情展示叶子主题，不再用 `文旅体育 / 产业项目 / 政策治理` 这类父级组合代替主题结论。

## 不建议的方案

1. **只扩充 `config/topics.json`**：从 10 个加到 30 个仍是封闭枚举，只会推迟下一次失真。
2. **完全自由且直接发布**：会产生同义词碎片、名称漂移和不稳定 URL，无法支撑长期主题页。
3. **继续允许五个并列主题且不分主次**：主题页会被弱相关内容污染，读者无法理解文章到底属于什么。
4. **把版面名当主题**：`人物、悦读、读图、文化周刊` 是编辑形态，不是文章议题。
5. **用关键词或 embedding 直接替代语义判断**：可以辅助召回与归一，但不应越过原文证据和人工可审计的主题结论。

最终原则可以压缩成一句话：**第一次入库忠实描述文章，第二次归一服务长期目录，第三步产品规则决定它是否进入“按主题看海南”。**
