# AIHOT 前端逻辑与逆向提示词资料设计

## 目标

更新 `references/aihot/` 到 AIHOT 2.0 的当前公开状态，补齐手机端前端证据，并生成一篇把每块可见功能、截图、前端逻辑和逆向提示词对应起来的中文 Markdown 文档。

## 边界

- 只分析公开页面、公开 API / OpenAPI、浏览器可见状态和已下载的前端资源。
- “逆向提示词”是根据输出结果推导的学习模板，不声称是 AIHOT 的真实内部提示词。
- 搜索、筛选、收藏、已读、主题切换、导航和 Markdown 导出属于确定性前端逻辑；为这些功能提供高效的“开发/实现提示词”，不伪装成内容生成提示词。
- 摘要、正文清洗与翻译、事件聚类、代表信源选择、分类、标签、评分、精选、热点、推荐理由和日报编排属于可能的模型辅助流程；为这些功能提供结构化逆向提示词。
- 提示词遵循 GPT-5.6 官方建议：精简、一次一条规则、明确自主边界、明确长度与语气、固定输出格式、失败与停止条件可验证。

## 证据与截图

- 桌面端：`1440 × 1000`。
- 手机端：`393 × 852`。
- 必须覆盖首页热点、分类与时间线、搜索、收藏、详情页、日报、主题、主题切换、更新日志等能证明功能逻辑的状态。
- 每张采用的截图必须先人工查看，排除空白、加载中、裁切错误或错误页面。

## 文件结构

- `references/aihot/raw/`：当前公开 HTML、样式表和与手机端/主要交互相关的前端脚本。
- `references/aihot/screenshots/`：按功能和视口命名的当前截图。
- `references/aihot/analysis/latest-changes.md`：从旧快照到当前状态的可验证变化。
- `references/aihot/analysis/reverse-engineered-prompts.md`：最终截图与提示词文档。
- `references/aihot/README.md`、`manifest.json` 和现有分析文件：更新阅读顺序、日期、文件清单和过时结论。

## 验收

1. 文档中的每个主要功能都有截图或明确说明为何复用同一截图。
2. 每个提示词都包含目标、输入、约束、输出格式和失败/边界规则。
3. 推断提示词明确标注置信度和证据，不能被读成泄露的内部实现。
4. `manifest.json` 的文件大小与 SHA-256 和磁盘文件一致。
5. Markdown 内所有相对截图链接都能解析到实际文件。
6. 旧结论与 AIHOT 2.0 冲突时必须更新，而不是在文档中继续并存。
