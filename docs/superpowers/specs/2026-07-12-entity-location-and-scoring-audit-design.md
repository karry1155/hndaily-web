# 新闻实体地点标签与评分审计设计

## 目标

为进入 HN·HOT 的每条候选新闻增加可验证、可聚合的“谁、在哪里、做了什么”结构化信息，为后续人物聚合、行政区筛选和海南热力图提供稳定数据；同时修正首页 TOP4 的信息密度，并生成一份忠实反映当前评分实现与真实数据分布的 HTML 审计报告。本次不改变评分公式、权重或阈值。

## 范围

本次包含：

- 在模型语义输出中增加主体、受限地点选择、动作摘要及原文证据。
- 建立项目内版本化的海南行政区字典和严格解析器。
- 将标准化后的实体数据写入正式精选条目、公开索引和搜索索引。
- 回填现有精选历史数据；无法可靠识别时保留空数组，不猜测。
- 首页 TOP4 展示“主主体 · 主地点”，并压缩多余空白、垂直居中内容。
- 生成当前评分机制 HTML 报告，分析全部现有精选数据为何高度集中在 80 分。

本次不包含：

- 修改五维评分权重、阈值、去重规则或精选数量。
- 实现地图、热力图、人物详情页或地点筛选页。
- 对主体名称建立跨新闻人物身份库或别名合并系统。

## 提取与标准化架构

采用“模型提取 + 脚本标准化”的双层方案。

模型读取标题和正文，并接收脚本根据正文命中结果生成的地点候选集合。模型只能从候选集合中返回 `location_id`，不能生成行政区名称、代码或层级。脚本根据本地字典补全标准字段。名单外 ID、缺失证据、未知主体类型或额外字段均使模型输出校验失败。

主体不使用封闭名称白名单，因为人物和机构会持续变化；主体类型使用封闭枚举，名称和角色必须有正文证据。脚本验证证据为正文中的非空片段，并限制长度。无法确定主语或地点时返回空数组。

## 海南行政区字典

新增版本化数据文件，至少覆盖海南省、市、县级行政区，并保留未来扩展乡镇级的能力。每项精确包含：

```json
{
  "location_id": "hainan-sansha",
  "name": "三沙市",
  "code": "460300",
  "level": "prefecture",
  "aliases": ["三沙"]
}
```

允许的 `level` 为 `province`、`prefecture`、`county`。行政区划代码在实现时使用权威来源核验，并在字典元数据中保存来源、核验日期和版本。代码升级必须显式更新版本，不能由模型修改。

地点候选脚本以规范名称和别名匹配标题、正文；模型只能选择脚本提供的候选。新闻只写“海南”时可选择省级地点；同时出现上级和下级行政区时，模型优先返回能由正文支持的最具体地点，并可保留多个地点。

## 模型输出契约

每个语义条目增加以下精确字段：

```json
{
  "actors": [
    {
      "name": "冯飞",
      "type": "person",
      "role": "省委书记",
      "evidence": "省委书记冯飞在三沙市调研"
    }
  ],
  "location_mentions": [
    {
      "location_id": "hainan-sansha",
      "evidence": "省委书记冯飞在三沙市调研"
    }
  ],
  "action": "调研学习教育及经济社会发展情况",
  "action_evidence": "在三沙市调研树立和践行正确政绩观学习教育及经济社会发展"
}
```

`actors` 最多 5 项，顺序表示新闻中的主体重要性。`type` 只能为 `person`、`organization`、`government`、`company`。`name`、`role` 和 `evidence` 必须是简短中文字符串；角色不明确时 `role` 为 `null`。

`location_mentions` 最多 5 项，只能使用当前候选集合中的 `location_id`，顺序表示地点重要性。`action` 是不超过 60 个汉字的事实性动作摘要；不存在可靠动作时为空字符串，`action_evidence` 同时为空字符串。

模型仍不得返回 `final_score`、行政区名称、行政区代码、精选决定、排名、URL 或来源。

## 正式存储结构

正式精选条目 schema 升级，新增顶层 `entities`：

```json
{
  "entities": {
    "actors": [
      {
        "name": "冯飞",
        "type": "person",
        "role": "省委书记",
        "evidence": "省委书记冯飞在三沙市调研"
      }
    ],
    "locations": [
      {
        "location_id": "hainan-sansha",
        "name": "三沙市",
        "code": "460300",
        "level": "prefecture",
        "evidence": "省委书记冯飞在三沙市调研"
      }
    ],
    "action": "调研学习教育及经济社会发展情况",
    "action_evidence": "在三沙市调研树立和践行正确政绩观学习教育及经济社会发展"
  }
}
```

名称、代码和层级完全由脚本从行政区字典补全。公开精选摘要、日期 feed、焦点和搜索索引传播 `entities`，确保动态滚动加载和收藏页使用同一数据。新闻全文页可保留完整实体数据，但本次不新增可视区块。

## 历史数据回填

现有精选历史条目使用同一模型提示、候选生成、严格校验和脚本标准化流程回填，不能针对四条首页新闻硬编码标签。回填必须保持原有标题、正文、URL、评分、排名和推荐理由不变。

验收样例：

- 第一条首页重点显示 `冯飞 · 三沙市`。
- 第二条显示 `刘小明 · 白沙县`。
- 第三条显示 `省政协 · 海口市`；页面文案可按产品短名显示为“海口”。
- 第四条如果正文不能同时支持主体和地点，则不显示实体标签。

若只识别出主体或只识别出地点，页面显示已识别的一项，不补占位符。

## 首页展示与密度

TOP4 每条标题下方增加灰色 `.story-entities` 文本，格式为 `主主体 · 主地点`。没有实体信息时不渲染该行。

焦点行使用三列网格：排名、正文、右侧评分与收藏。正文区域在行内垂直居中；有实体行时标题和实体作为紧凑的两行内容，无实体行时标题单行居中。减少上下内边距和无意义空白，但保留两行标题截断能力。桌面与手机均显示实体文本，字号低于标题，颜色使用现有 `--muted`。

普通新闻列表不增加实体显示，以免推荐理由、评分和收藏区域过密；实体仍存在于 JSON 和搜索索引中。

## 当前评分机制 HTML 审计报告

新增独立报告 `docs/scoring-system-audit.html`，只读分析当前实现，不提出或应用新评分公式。报告包含：

1. 从模型五维语义分到 `base_score`、调整项、`final_score`、入选阈值、日内排名和首页焦点分的完整流程图。
2. 每一步明确标注“模型判断”或“脚本计算”，链接对应源码文件和函数名。
3. 当前五个维度、权重、取值范围、舍入方式、阈值、同分排序和跨日衰减公式。
4. 对全部现有精选条目的真实统计：各维度分布、基础分和最终分分布、唯一分值数量、80 分占比、按日期分布及同分情况。
5. 选取若干真实条目展示从模型原始五维分到最终分的逐项计算。
6. 明确诊断“大量 80 分”的直接原因，区分模型输出同质化、公式离散度、调整项缺失或数据回填方式等因素。
7. 列出下一轮重建评分系统时需要决策的问题，但不预先选定答案。

报告必须为自包含 HTML，使用现有品牌颜色，支持桌面和手机阅读，不依赖外部 CDN。

## 错误处理

- 地点字典重复 ID、重复代码、非法层级或缺少元数据时构建失败。
- 模型返回候选外地点、超过数量限制、非法主体类型或额外字段时整批失败，不静默丢弃。
- 证据不在标题或正文的规范化文本中时失败。
- 历史回填无法确认实体时写入空数组和空动作，不生成猜测标签。
- 页面渲染对缺失实体保持兼容，旧索引在迁移期间不会崩溃。

## 测试与验收

- 单元测试覆盖行政区字典、别名命中、候选限制、模型严格字段、证据校验和正式结构补全。
- 流水线测试覆盖实体从模型输入到正式条目、索引、动态 feed 和页面的完整传播。
- 真实数据验收覆盖三条指定首页新闻及第四条空标签行为。
- 页面测试覆盖 TOP4 垂直居中、紧凑间距、实体文案和无实体时不留空行。
- 报告测试校验真实统计与源 JSON 一致，并检查关键公式、源码位置和诊断章节存在。
- 全量测试、静态站构建和浏览器 592px 手机视口截图校验后才可提交完成。
