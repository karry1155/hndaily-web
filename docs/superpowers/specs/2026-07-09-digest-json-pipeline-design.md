# 海南日报 JSON 精读流水线设计

## 背景

现有爬虫能够稳定生成包含版面、标题、正文和原文链接的原始 JSON，但抓取后的生产流水线没有日报生成器。`run_daily_pipeline.sh` 在抓取完成后只输出下一步提示，网站当前展示内容来自手工 fixture，并非真实日报。因此本轮只补齐从原始 JSON 到可发布日报 JSON 的最短链路。

## 目标

- 从爬虫原始 JSON 中读取文章，生成可供现有网页渲染的日报精读 JSON。
- 每条精读包含可读标题、内容提炼、值得阅读的原因、关键事实和原始链接。
- 语义分析由 Codex 定时任务完成，项目不保存或调用独立模型 API Key。
- 生成结果必须落盘、可校验、可重复运行，并能接入后续 Git 提交和 GitHub 推送。

## 本轮范围

本轮实现验证模式：按照原始 JSON 中的版面顺序和版内 `seq` 顺序固定取前三篇文章。固定输入便于比较提示词和输出变化，也避免验证阶段处理整份日报。

本轮不实现：

- 全量日报的自动价值排序和最多五篇精选。
- 长期资料库和跨日主题追踪。
- 图形化编辑反馈后台。
- 读者点赞、点踩或行为分析。
- 项目代码直接调用外部模型 API。

## 方案

采用“确定性脚本 + Codex 语义处理 + 确定性校验”的轻量方案：

1. `crawler.py` 生成原始日报 JSON。
2. 准备脚本读取所有版面文章，按版面和 `seq` 排序，固定提取前三篇并写入候选 JSON。
3. Codex 定时任务读取候选 JSON，逐篇提炼新闻内容，严格生成指定结构的日报 JSON。
4. 校验脚本检查结构、来源可追溯性和前三篇覆盖关系。
5. 校验通过后，现有渲染脚本生成静态网站；定时任务再执行 Git 提交和推送。

不让 Codex 直接修改原始数据。原始 JSON、候选 JSON、Codex 原始输出和最终日报 JSON 分层保存，便于定位错误和比较不同版本。

## 数据流

```text
原始 JSON
  -> 固定提取前三篇
  -> 候选 JSON
  -> Codex 语义提炼
  -> 日报 digest JSON
  -> 结构与来源校验
  -> 静态页面渲染
  -> GitHub 推送
```

候选文章必须保留：

- `article_id`：由日期、版面和文章序号组成的稳定标识。
- `page`：原始版面号。
- `seq`：版内顺序。
- `headline`：原始标题。
- `content`：原始正文。
- `url`：原始链接。

Codex 为每篇生成：

- `title`：基于正文改写的可读标题。
- `summary`：发生了什么，只陈述原文支持的事实。
- `why_it_matters`：为什么用户今天值得关注。
- `key_facts`：日期、数字、地点、主体和动作组成的事实列表。
- `sources`：原始标题、版面和 URL，不得改写或补造。
- `confidence`：根据正文完整性标记为 `full_text`、`short_item` 或 `partial`。

最终产物继续兼容现有 `content/daily/YYYY-MM-DD.json` 和页面渲染器。前三篇分别进入 `top_items`，其他固定分类在验证阶段允许为空。

## Codex 定时任务边界

Codex 负责：

- 执行爬虫和候选准备命令。
- 阅读候选文章正文。
- 按项目中的提示词和数据契约生成 digest JSON。
- 执行校验、渲染、Git 提交和 GitHub 推送。

项目脚本负责：

- 固定选择前三篇。
- 生成稳定的候选结构。
- 验证字段、URL、版面、文章标识和来源覆盖。
- 保证校验失败时不进入渲染和推送步骤。

## 幂等与故障处理

- 同一原始 JSON 内容对应同一候选输入指纹。
- 已存在相同输入指纹的合格 digest 时默认复用，除非显式要求重新生成。
- 少于三篇时处理全部可用文章，不补造条目。
- 任何候选文章缺少标题、正文或 URL 时，准备步骤明确报错并停止。
- Codex 输出不是合法 JSON、来源 URL 不匹配、条目遗漏或出现额外来源时，校验失败。
- 校验失败不得覆盖上一份合格页面，也不得自动提交或推送。

## 反馈演进基础

本轮不建设反馈界面，但为后续迭代保留：

- 候选输入指纹。
- 提示词版本。
- Codex 原始生成结果。
- 最终发布结果。

未来可以在不改变流水线边界的情况下增加人工的保留、降级、删除和改写记录，并把这些记录转化为提示词示例和回归用例。

## 测试与验收

- 给定真实的 `2026-07-08.json`，候选文件严格包含全局顺序的前三篇文章。
- 重复运行候选准备命令产生相同文章顺序和输入指纹。
- 三篇文章都生成 `title`、`summary`、`why_it_matters`、非空 `key_facts` 和原始来源。
- 每个 digest 来源 URL、标题和版面都能在候选 JSON 中精确找到。
- 缺失、改写或虚构来源会导致校验失败。
- 合格 digest 能通过现有数据验证并由现有渲染器生成页面。
- 验证测试只处理固定前三篇，不对其余文章发起语义分析。

## 后续阶段

验证模式稳定后，再单独设计全量候选过滤、同主题聚类和最多五篇价值排序。本轮实现不会提前加入这些逻辑。
