# HNHOT 产品与数据基础设计

## 1. 文档状态

- 日期：2026-07-18
- 状态：产品方案已确认，等待书面规范复核
- 适用仓库：`hndaily-web-radar`
- 正式产品名：HNHOT
- 首期信源：海南日报
- 实施基线：当前 `main` 分支

本文定义 HNHOT 的产品定位、移动端信息架构、头版分类、全报阅读、长期归档、AI 加工契约、项目内数据布局、自包含抓取边界、错误处理、测试要求和实施顺序。它取代此前以“AI 精选评分”和“全量文章展示”为核心的产品定义。

## 2. 产品定义

HNHOT 不是新闻聚合器，不为海南日报再做一次人工或 AI 精选，也不是只有展示能力的电子报阅读器。

HNHOT 放大海南日报编辑已经做出的专业判断，通过 AI 将每日报道结构化、关联化和长期归档，帮助用户在持续阅读中形成对海南人物、机构、地区、议题和事件的个人理解。

产品价值链为：

> 看见重要信息 → 快速理解 → 收藏或继续追踪 → 建立连续记忆 → 形成对海南的长期认知

阅读是入口。用户花费阅读时间后形成的收藏、主体档案、地区档案和事件时间线，才是产品长期价值。

## 3. 前期方向复盘与代码处置

### 3.1 `human-feedback-review`

该分支通过人工反馈、标注和再次精选提高内容质量，但忽略了海南日报本身已经经过专业编辑筛选和版面编排。HNHOT 不再建设第二套编辑体系。

处置规则：

- 不整体合并人工审核、评分后台和复杂发布流程。
- 不保留人工标注作为日常内容生产的必经步骤。
- 仅在实施时按需复用与业务方向无关的通用能力，例如提示词版本管理、JSON 严格校验和原子写入。

### 3.2 `modern-reader-core`

该分支解决了现代化展示整份报纸的问题，但没有形成阅读后的长期积累。HNHOT 不以“完整展示”作为终点。

处置规则：

- 不整体合并该分支。
- 按需复用全量文章数据契约、正文详情、搜索、归档和静态发布能力。
- 新实现必须把每日阅读与收藏、主体、地区和事件时间线连接起来。

两个失败分支保留为历史参考，实施以当前 `main` 为基线，不从任一分支继续演进。

## 4. 产品原则

1. **尊重报纸的编辑判断。** 头版、版面位置和原始顺序本身就是编辑信号，不使用 AI 分数替代。
2. **AI 负责增益，不负责重新办报。** AI 用于摘要、分类、实体提取和事件关联，不决定一篇报道是否值得被公开。
3. **完整性优先。** “全部”保留所有成功抓取的有效报道，不通过评分阈值删除内容。
4. **来源可追溯。** 摘要、主体、地区、议题和事件关联必须保存原文证据并能返回来源报道。
5. **长期积累优先。** 每日加工既服务当日页面，也必须持续更新主体、地区和事件索引。
6. **保守关联。** 无法确认的主体、地区或事件关系保持为空；宁可暂时不连接，也不建立错误连接。
7. **静态和本地优先。** 保留本地生成、静态发布架构，首期不引入公网写入后台。
8. **仓库自包含。** 抓取器、提示词、JSON Schema、字典、脚本、测试和操作说明全部位于当前仓库。

## 5. 信息架构与路由

移动端底部固定四个一级入口：

| 入口 | 路由 | 职责 |
|---|---|---|
| 头版 | `/` | 放大海南日报头版承载的编辑判断 |
| 全部 | `/all/` | 按归并后的报纸版面阅读完整内容 |
| 日报 | `/daily/` | 未来的当日综合阅读产物，首期仅保留空状态 |
| 更多 | `/more/` | 进入收藏、主体档案和地区档案 |

二级路由包括：

- `/starred/`：收藏。
- `/subjects/`：按主体看海南。
- `/subjects/{subject_id}/`：主体详情与时间线。
- `/regions/`：按地区看海南。
- `/regions/{location_id}/`：地区详情与报道列表。
- `/stories/{event_id}/`：事件时间线。
- `/items/{item_id}/`：报道详情与原文核对。

对外品牌、页面标题和公开文案统一使用“HNHOT”。仓库目录在本阶段继续保留 `hndaily-web-radar`，避免破坏工作树、脚本和部署配置。仓库更名属于独立的运维变更，不进入本轮产品实现。

## 6. 头版

### 6.1 页面目标

“头版”不是 HNHOT 自己生成的精选，而是一个持续积累的海南日报头版信息池。页面放大三类编辑价值：一条报道是否进入头版、报道在头版中的原始顺序、头版内容与海南之间的关系。

### 6.2 页面头部

页面标题统一为“头版”。顶部保留：

- HNHOT 品牌。
- 当前报纸日期及日期切换。
- 搜索。
- “全部”“全国”“海南”“海南关联”四个筛选项。

### 6.3 “全部”视图

默认视图按出版日期倒序显示全部头版报道。当日内容分为两个层级。

第一层是“全国要闻 TOP N”：

- 只包含当日分类为 `national` 的头版报道。
- `N` 根据当日实际数量动态变化，不补足、不截断为固定数量。
- 排名严格沿用报纸头版中的原始文章顺序，不使用 AI 重要性评分。
- 当日没有全国类报道时，整个排行榜模块隐藏。

第二层是完整头版信息流：

- 展示当日全部 `national`、`hainan`、`mixed` 报道。
- 每条信息显示 N/H/M 标记、原始标题和 AI 摘要。
- 已进入排行榜的全国类报道仍在完整信息流中保留；排行榜负责强化，信息流负责完整。
- 历史日期继续按天分组向下排列。

### 6.4 分类视图

- “全国”按日期展示全部 `national` 类头版报道，不显示排行榜。
- “海南”按日期展示全部 `hainan` 类头版报道，不显示排行榜。
- “海南关联”按日期展示全部 `mixed` 类头版报道，不显示排行榜。

筛选只改变浏览视图，不改变底层内容是否存在。

### 6.5 N/H/M 分类契约

存储使用英文枚举，界面使用单字母标记：

| 存储值 | 标记 | 定义 |
|---|---|---|
| `national` | N | 主要主体和行动属于全国性事务，正文没有海南机构、人物、项目、活动或直接影响参与其中 |
| `hainan` | H | 主要行动、事件或治理对象发生在海南，核心意义属于海南本地治理、经济、民生、社会或文化事务 |
| `mixed` | M | 同时存在全国或国际层面的背景，以及可从原文核对的海南人物、机构、企业、项目、赛事或活动连接 |

判定顺序为：

1. 同时具有外部议程和明确海南连接时归为 `mixed`。
2. 否则，主要行动或事件发生在海南时归为 `hainan`。
3. 其余全国性内容归为 `national`。

AI 返回分类和原文证据，脚本验证枚举、证据和文章对应关系。分类不产生分数。

## 7. 全部

“全部”负责完整阅读当期报纸，不进行二次精选。

### 7.1 版面归并

报纸因为纸面篇幅限制，可能把同一栏目拆分到多个物理版面。例如 002、003、004 均可属于“本省新闻”。HNHOT 将它们归并为一个“本省新闻”分组，前台不显示 `002`、`003`、`004` 等物理版号。

归并规则：

- 使用仓库内版本化的 `config/page-sections.json`，不让 AI 临时猜测。
- 同一归并版面内继续按照原物理版号和版内文章顺序排列。
- 物理版号、原始版名、PDF 地址和版面链接继续保存在数据中，供详情页核对。
- 无法匹配规则的版面使用原始版名，不阻断整期发布。
- 配置校验拒绝重复映射、空名称和指向不存在规范分组的规则。

### 7.2 内容展示

每个归并版面显示规范版面名称、报道标题、AI 摘要和详情入口。不显示推荐分、精选状态、推荐理由或人工标签。

详情页保留原始标题、日期、物理版号、原始版名、作者、与列表相同的唯一 AI 摘要、抓取正文、原文链接及同一期上一篇/下一篇导航。

## 8. 日报

“日报”保留一级路由，但首期不开发生成能力。页面只提供明确空状态，不生成示例数据，也不建立独立流水线。

日报后续必须建立在已经稳定的全量摘要、主体、地区、议题和事件数据之上，并通过单独设计后实施。

## 9. 更多与长期认知资产

### 9.1 收藏

收藏对象包括单篇报道、主体、地区和事件时间线。首期使用本地存储，数据结构必须带对象类型、稳定 ID 和收藏时间，并保留后续导入、导出或账号同步的扩展能力。

### 9.2 按主体看海南

主体是能够在多篇报道中持续出现、值得形成时间线的稳定对象。首期类型为：

- `person`：人物。
- `government`：政府部门。
- `organization`：公共机构或其他组织。
- `company`：企业。
- `project`：项目。

主体目录不把全部对象平铺成无限增长的卡片墙。默认提供搜索，并支持按主体类型、最近有新报道、报道数量和收藏状态浏览。

主体使用“规范名称 + 别名”机制。AI 提取原文名称、类型、角色和证据；脚本匹配已有主体。无法可靠匹配时保留为未解析提及或创建独立候选，不能仅凭相似名称强行合并。

主体详情页按时间展示发生日期、当次报道新增进展、涉及地区和议题、关联事件及来源报道。

### 9.3 按地区看海南

地区使用仓库内固定、版本化的海南行政区字典。模型只能选择脚本依据正文生成的候选 `location_id`，不能自由生成行政区名称和代码。

首期覆盖海南全省、地级市和县级行政区。地区详情页按日期展示相关报道，并链接相关主体和事件。

地区报道数量只表示海南日报在指定范围内的报道量，不代表地区重要性、发展水平或风险程度。无法可靠定位的报道不强行落点。

### 9.4 事件时间线

单篇报道是不可替代的原始记录；事件时间线是在文章之上建立的关联层。一个事件对象至少包含：

- 稳定 `event_id`。
- 规范事件名称。
- 首次和最近报道日期。
- 相关主体、地区和议题。
- 按日期排序的报道节点。
- 每个节点相对于前次报道出现的新进展。
- 全部来源文章 ID。

事件关联采用保守策略：

- 检索脚本先根据主体、地区、议题和文本相似性提供有限候选事件。
- 模型只能选择候选 `event_id` 或返回 `null`，不能自行生成已有事件 ID。
- 无法确认属于同一事件时创建新事件。
- 不因标题相似强行合并。
- 后续事件合并必须保留原始文章、旧事件 ID 映射和历史关联记录。

## 10. AI 内容加工契约

每篇有效报道统一生成：

- 一份全站复用的 `ai_summary`。
- `scope`：`national`、`hainan` 或 `mixed`。
- `subjects`：主体提及及证据。
- `locations`：候选行政区选择及证据。
- `topics`：受控议题 ID 及证据。
- `event_relation`：候选事件选择或新事件建议。

非头版文章也保存 `scope`，但 N/H/M 只在头版作为主要导航。

模型只输出语义信息。文章 ID、原始版面顺序、排行榜顺序、规范主体 ID、事件 ID、地区名称与行政区代码、索引和发布路径全部由脚本确定。

模型输出使用严格 JSON Schema：

- 所有对象启用 `additionalProperties: false`。
- 分类、主体类型和事件关系使用封闭枚举。
- 地区只能选择当前文章输入中提供的候选 ID。
- 议题只能选择版本化议题表中的 ID。
- 主体名称、地区、议题和事件判断均保存正文证据。
- 不确定时返回空数组或 `null`，不得猜测。
- 输出保存 `schema_version`、`prompt_version` 和 `input_fingerprint`。
- 输出条目数量、顺序和 `item_id` 必须与输入完全一致。

摘要要求：

- 一般为 1–3 句，压缩会议流程、礼仪性动作、空泛表态和宣传口号。
- 保留具体行动、地点、数字、时间、政策变化、项目进展和下一步安排。
- 不新增评价、影响推测或“值得关注”等模板化结论。
- 所有事实可在输入正文中找到依据。
- 正文缺失时不根据标题猜测摘要，输出空摘要并由页面显示缺失状态。

```text
hndaily-web-radar/
├── scripts/
│   └── crawler.py
├── config/
│   ├── page-sections.json
│   ├── subjects.json
│   └── topics.json
├── prompts/
│   └── article-enrichment/
│       └── v1/
│           ├── prompt.md
│           ├── schema.json
│           └── manifest.json
├── data/
│   └── json/
│       ├── raw/
│       │   └── YYYY-MM-DD.json
│       ├── model-input/
│       │   └── YYYY-MM-DD.json
│       ├── model-output/
│       │   └── YYYY-MM-DD.json
│       └── audits/
│           └── YYYY-MM-DD.json

    ├── issue-items/
    └── indexes/
        ├── front-page/
        ├── subjects/
        ├── regions/
        └── stories/
```

目录职责：

- `data/json/raw/` 保存爬虫原始输出，按日期落盘，不经过 AI 修改。
- `data/json/model-input/` 保存实际交给模型的完整输入。
- `data/json/model-output/` 保存严格结构化模型输出。
- `data/json/audits/` 保存抓取缺失、校验失败、分类证据和关联结果。
- `content/` 保存通过校验、可供静态站发布的正式数据。
- `prompts/` 保存版本化提示词、机器可验证 JSON Schema 和版本清单。

原始数据和模型中间产物位于项目目录但默认加入 `.gitignore`。提示词、Schema、配置、测试夹具和公开内容进入版本控制。

## 12. 抓取器内聚

将 `hndaily-skill/crawler.py` 的核心能力迁入 `scripts/crawler.py`。迁移后的抓取器：

- 默认把原始数据写入 `data/json/raw/`。
- 不读取 `HNDAILY_SKILL_DIR`，不引用 `hndaily-skill` 中的任何文件。
- 依赖仅使用 Python 标准库或在当前仓库显式声明。
- 保留指定日期、自动发现最新一期、30 分钟缓存、强制重抓、并发抓取和错误记录能力。
- stdout 第一行继续输出生成 JSON 的绝对路径，便于流水线组合。
- 为 HTML 解析、版面顺序、文章顺序、空正文和网络失败提供仓库内测试夹具。

README 必须给出从抓取、生成模型输入、写入模型输出、校验、发布到预览的完整命令。验收标准是任意智能体只进入当前仓库，无需了解相邻目录，即可独立开展工作。

## 13. 数据流与发布事务

每日处理流程为：

1. 抓取海南日报，原子写入 `data/json/raw/YYYY-MM-DD.json`。
2. 解析并规范化所有有效文章，保留原始版面和版内顺序。
3. 根据行政区字典、主体库、议题表和近期事件库生成受限候选。
4. 写入带输入指纹的 `model-input`。
5. 模型依据仓库内提示词生成严格 `model-output`。
6. 脚本校验字段、顺序、证据、候选限制和输入指纹。
7. 脚本解析规范主体、地区、议题和事件关系，生成审计记录。
8. 在临时目录构建完整 `content` 代次及所有索引。
9. 全量验证成功后原子替换正式代次并渲染静态站。

任何步骤失败都不得留下部分更新的公开内容。相同日期重跑采用“同源同日期整体替换”，同时保留其他日期。相同输入和模型输出必须产生确定性相同的正式数据。

## 14. 降级与错误处理

- 当日没有报纸版面时记录明确状态，不生成伪造空期。
- 单个版面抓取失败时保留版面错误并继续其他版面；整期审计标记为不完整。
- 单篇正文失败时保留标题、来源和错误，不根据标题生成摘要。
- 模型输出缺项、多项、乱序、额外字段、非法枚举、候选外 ID 或无效证据时整批拒绝。
- 单篇摘要为空不阻断整期发布，页面显示“摘要待生成”或“正文抓取失败”。
- 主体无法解析时保留原始提及，不合并到相似主体。
- 地区无法确认时不写地点。
- 事件无法确认时创建新事件或保持未关联，不强行合并。
- 版面归并规则缺失时回退到原始版名。
- 任一正式内容或索引校验失败时回滚整个发布代次。

## 15. 测试与验收

### 15.1 数据与抓取

- 单元测试覆盖版面列表、文章链接、正文解析、日期发现、缓存、强制重抓和网络失败。
- 契约测试覆盖原始 JSON、模型输入、模型输出、正式文章、期次和索引。
- 提示词测试校验 manifest、prompt 和 JSON Schema 版本一致。
- 自包含测试确保命令和代码不引用 `hndaily-skill`、`HNDAILY_SKILL_DIR` 或仓库外路径。

### 15.2 头版与全部

- 验证头版只读取物理头版文章。
- 验证 TOP N 数量随当日全国类报道数量变化，顺序与原报一致。
- 验证进入 TOP N 的报道仍出现在完整信息流。
- 验证三个分类视图不显示排行榜。
- 验证版面归并后不展示 `00x` 物理版号，文章顺序保持稳定。
- 验证无法归并时使用原始版名。

### 15.3 长期索引

- 验证主体规范名称、别名、未解析提及和错误合并保护。
- 验证地区只能来自行政区候选集合。
- 验证议题只能来自受控词表。
- 验证事件只能选择候选 ID 或创建新事件。
- 验证同一日期重跑不会产生重复主体、事件或时间线节点。

### 15.4 界面与发布

- 移动端验证四个固定一级入口及当前路由状态。
- 桌面端和移动端验证头版、全部、更多、详情和空日报页面。
- 验证搜索与收藏使用稳定文章 ID。
- 验证模型或单篇正文失败时的可理解降级状态。
- 全量测试、静态站构建和代表性移动端截图检查通过后才能宣告实施完成。

## 16. 实施顺序

采用“数据基础与页面垂直切片结合”的方式：

1. 确立 HNHOT 数据契约，迁入自包含抓取器，建立 JSON 目录和版本化提示词。
2. 跑通“抓取一日 → 全量 AI 加工 → 严格校验 → 正式发布”的最小闭环。
3. 实现头版，完成 N/H/M 分类、全国要闻排行榜和按日信息流。
4. 实现全部，完成版面归并和全量阅读。
5. 在每日加工中同步积累主体、地区、议题和事件索引。
6. 实现更多中的收藏、按主体看海南、按地区看海南和事件时间线。
7. 保留日报空路由，待基础数据稳定后单独设计。

不采用“先完成所有页面再补数据”的路径，避免再次形成只有展示能力的阅读器；也不采用“先建设完整知识图谱再开放页面”的路径，避免长期没有真实产品反馈。

## 17. 首期非目标

- 人工打标和人工精选后台。
- AI 新闻重要性评分。
- 固定数量的精选内容。
- 多信源聚合。
- 自动生成日报。
- 复杂账号系统。
- 对主体或地区做价值排名。
- 高风险主体或事件自动合并。
- 无原文依据的摘要、标签或推断。

## 18. 首期完成标准

- 产品内统一显示 HNHOT。
- 四个移动端一级路由稳定存在。
- 头版完整展示并能按 N/H/M 筛选。
- 全国要闻 TOP N 沿用原报顺序且数量动态变化。
- 全部页面完成版面归并，列表不暴露 `00x` 物理版号。
- 每篇有效报道都有唯一摘要，并尝试提取主体、地区和议题。
- 所有模型输出通过严格 JSON Schema 校验。
- 抓取和加工不依赖仓库外文件。
- 原始抓取按日期落入项目数据目录。
- 主体、地区和事件索引随每日报道持续增长。
- 任意智能体可仅凭仓库 README 独立运行完整流程。
