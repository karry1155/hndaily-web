# 个人信息雷达：精选库、分类视图与详情页设计

## 1. 文档状态

本文档是对 `2026-07-10-hainan-editorial-ranking-design.md` 的产品方向升级。旧设计记录了当前 `editorial-v1` 的实现背景；当两份文档冲突时，本设计优先。

已确认的核心方向：网站的主产品是持续增长的个人信息雷达，不是《海南日报》的另一种呈现。《海南日报》只是当前第一个信源；未来信源通过适配器进入同一套信息块、评分、分类、精选和展示流程。

## 2. 目标与非目标

### 2.1 目标

- 把每篇信源文章规范化为稳定、信源无关的信息块。
- 对每个信息块独立生成摘要、评分和唯一分类。
- 将所有达到资格线的内容永久保存到精选库，不设置每日入选数量上限。
- 首页以“当下重点”突出最近内容中的 Top 4，同时保持历史精选可浏览。
- 为“全部、机会、民生、产业、政策、城市、观察”提供符合各自阅读目的的视图。
- 用日期分区、分页和索引保证精选库持续增长后仍可快速生成和阅读。
- 让热点和信息卡进入站内详情页，在详情页同时提供 AI 摘要、完整原文正文和原始链接。
- 用与未来定时任务相同的增量入口抓取并处理新日期，证明流水线不依赖手工准备的固定样本。

### 2.2 本轮非目标

- 不新增海南日报以外的实际信源，只建立适配接口并迁移现有爬虫输出。
- 不做跨信源事件聚合。海南日报内部按编辑成稿处理，不额外执行当前的近重复聚类。
- 不建设多标签体系、标签聚合页、全文搜索、收藏、用户账号或人工编辑后台。
- 不用模型改写信源标题，不生成 `hot_title`。
- 不引入分钟级时间线或实时新闻承诺。
- 不部署、推送或创建 PR。

## 3. 方案选择

### 3.1 方案 A：统一精选池加确定性视图（采用）

所有信源先适配为统一信息块。模型只生成摘要、分类和语义评分；代码决定资格、排名、当下重点、机会生命周期和页面索引。首页、分类页、日期归档、日报和周报都是精选库的确定性视图。

该方案把信源接入、语义判断、编辑决策和页面展示分开，最适合后续增加信源，并能沿用当前严格模型契约、输入指纹和审计机制。

### 3.2 方案 B：两阶段模型精选

先让模型挑选内容，再二次生成热点和分类。它更灵活，但调用成本更高、结果波动更大，也更难验证所有来源是否被完整评估。本轮不采用。

### 3.3 方案 C：只修改当前日报页面

隐藏版面和分数、增加选项卡，但保留日报为主产物和每日 8 条上限。该方案没有解决信源扩展和精选库增长问题，不采用。

## 4. 总体架构

```text
海南日报 crawler.py                 未来信源抓取器
        │                                  │
        ▼                                  ▼
HainanDailyAdapter                   FutureSourceAdapter
        └──────────────┬───────────────────┘
                       ▼
              标准来源候选记录
                       ▼
          全量校验、摘要、评分、唯一分类
                       ▼
            完整信息块与当日精选结果
                       ▼
                  持久精选库
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
  全部/分类视图     当下重点 Top 4    详情页/衍生日报周报
```

### 4.1 模块职责

- 信源抓取器：获取信源原始数据，不做网站级精选。
- 信源适配器：把各信源字段转换为标准来源候选记录，注入可信标题、正文、URL 和日期。
- 模型交接层：只提交标题和正文，接收摘要、唯一分类、语义评分和机会生命周期证据。
- 编辑核心：校验模型输出，计算最终分，决定入选和稳定排名。
- 精选存储：按发布日期持久保存每个入选条目。
- 索引构建器：生成最新内容、分类、日期、机会有效状态和分页索引。
- 静态渲染器：从精选条目与索引生成首页、分类页、日期页和详情页。

各模块通过普通字典或已校验 JSON 交接。信源适配器之外的模块不得依赖报纸版面号、版面名或 `seq`。

## 5. 统一信息块与编辑记录

### 5.1 稳定核心信息块

模型处理和代码合并完成后，每个公开信息块必须具有以下五个稳定字段：

```json
{
  "source": "海南日报",
  "title": "海南启动2026年大学生科技见习计划",
  "content": "信源抓取到的完整正文",
  "ai_summary": "根据正文生成的摘要",
  "original_url": "http://news.hndaily.cn/..."
}
```

字段所有权：

- `source`、`title`、`content`、`original_url` 由代码从信源适配器注入。
- `ai_summary` 由模型根据标题与正文生成。
- 模型不得返回或覆盖 `source`、`title`、`content`、`original_url`。
- 首页、详情页和未来衍生内容都消费同一信息块，避免为某个页面再造内容字段。

### 5.2 外层编辑元数据

增长、分类和审计所需字段放在信息块外层：

```json
{
  "schema_version": 3,
  "item_id": "hndaily-2026-07-08-a018",
  "published_date": "2026-07-08",
  "collected_date": "2026-07-10",
  "category": "机会",
  "semantic_scores": {
    "hainan_relevance": 10,
    "actionability": 9,
    "impact_scope": 6,
    "timeliness": 8,
    "information_density": 10
  },
  "final_score": 86.5,
  "selected": true,
  "daily_rank": 4,
  "opportunity": {
    "lifecycle": "dated",
    "deadline_date": "2026-07-10",
    "deadline_text": "岗位征集截至7月10日",
    "evidence": "正文中的原句"
  },
  "block": {
    "source": "海南日报",
    "title": "海南启动2026年大学生科技见习计划",
    "content": "完整正文",
    "ai_summary": "AI 摘要",
    "original_url": "http://news.hndaily.cn/..."
  }
}
```

`item_id` 必须由信源标识和信源内稳定标识确定，重复运行不得变化。海南日报适配器优先使用原始 URL 中稳定的内容 ID；只有无法取得时才使用日期与规范化标题的哈希。

相同 `item_id` 的重复运行执行幂等更新，不产生第二条内容。若摘要、分类或评分因明确的版本升级发生变化，覆盖当前条目并在审计记录中保留旧版本号和生成版本；原始标题、正文和 URL 仍以信源适配器结果为准。

`published_date` 是页面排序和归档日期。海南日报使用报纸出版日期。系统不要求也不公开时分秒。

## 6. 模型职责与严格契约

### 6.1 模型输入

模型输入包含：

- `candidate_id`
- `title`
- `content`

模型输入不包含 URL、版面、作者、最终分、入选状态或排名规则。

### 6.2 模型输出

每个候选必须且只能返回：

- `candidate_id`
- `ai_summary`
- `category`
- 五个 0–10 整数语义分
- 五个评分理由
- `opportunity_lifecycle`
- 可选 `deadline_date`、`deadline_text` 和 `deadline_evidence`

`category` 只能是六个正式分类之一。`opportunity_lifecycle` 只能是：

- `dated`：正文有明确截止日期。
- `ongoing`：正文明确说明长期、常态或持续有效。
- `unspecified`：属于机会，但正文没有期限信息。
- `not_applicable`：非机会分类。

只有“机会”可以使用前三种生命周期；其他分类必须为 `not_applicable`。`dated` 必须同时提供三个截止字段，代码校验日期格式，并要求 `deadline_text` 或 `deadline_evidence` 能在规范化正文中找到依据。模型不得猜测未写明的期限。

输入指纹、schema、prompt 版本、数量、候选 ID 集合和顺序任一不匹配时，流水线失败。

## 7. 分类体系

首页与分类视图固定为：

- `全部`：视图，不是分类；展示精选库中的近期全部内容。
- `机会`：报名、招聘、征集、申报、补贴、活动参与、临近截止。
- `民生`：教育、医疗、住房、就业、社保、公共服务、娱乐。
- `产业`：项目、企业、园区、旅游、农业、科技和产业数据。
- `政策`：新规、监管、政务措施及其实际影响。
- `城市`：交通、环境、天气、公共安全、城市建设。
- `观察`：趋势、反腐、人事监督、重大问责，以及不适合归入以上类别的重要事件。

每条内容只能属于一个正式分类。分类依据是“读者为什么关注”，不是信源栏目、报纸版面或机构属性。信息卡不重复显示一级分类；当前选项卡和详情页负责提供分类上下文。

多标签体系本轮暂缓。未来可在外层编辑元数据增加多值 `tags`，但不得改变一级分类的互斥性。

## 8. 评分、精选与排序

### 8.1 信源无关的五维评分

保留当前五维加权结构，但把 `novelty` 明确升级为 `timeliness`：

- 海南相关性 `30%`
- 行动价值 `25%`
- 影响范围 `20%`
- 时效性 `15%`
- 信息密度 `10%`

基础分为五维加权结果乘以 10。入选资格保留当前双资格线：

- `hainan_relevance >= 6`
- `final_score >= 65`

取消头版、本省新闻版和版面顺序加分。短内容不会因为字数本身被硬删除；其信息是否充分由信息密度分表达。筛选核心不得依赖海南日报专有字段。

### 8.2 当日精选

- 所有同时通过双资格线的内容都进入精选库。
- 不设置每日最多 8 条或其他数量上限。
- 不设置分类配额，不为凑数降低资格线。
- 同一运行结果按最终分、信息密度、发布日期和稳定 ID 排序并获得 `daily_rank`。
- 当前版本不做报纸内部去重；未来多信源去重另行设计。

### 8.3 “当下重点”

“当下重点”只在“全部”视图出现，最多四条。候选池由精选库中最近三个有内容的自然日组成，而不是最近 72 小时，因此不需要分钟级时间戳。

代码使用确定性焦点分：

```text
focus_score = final_score - 3 × older_content_day_index
```

最新内容日的 `older_content_day_index` 为 0，次新为 1，第三新为 2。按焦点分、最终分、发布日期和稳定 ID 排序取前四。这样重要内容可以跨日保留，新内容也获得适度优先级。少于四条时如实显示实际数量。

每次新增精选条目后重建该索引。标题直接使用信源标题，不截断、不改写、不生成 `hot_title`。区域标注“更新至 YYYY-MM-DD”。

## 9. 精选库增长与索引

### 9.1 按日期分区

精选条目按发布日期单独保存，避免单个 JSON 无限膨胀：

```text
content/
  items/
    2026-07-08/
      hndaily-2026-07-08-a018.json
    2026-07-09/
      ...
  indexes/
    focus.json
    all/
      page-001.json
    dates/
      2026-07-08.json
    categories/
      opportunity/
        active-page-001.json
        expired-page-001.json
      livelihood/
        page-001.json
      industry/
        page-001.json
      policy/
        page-001.json
      city/
        page-001.json
      observation/
        page-001.json
```

条目文件是事实来源，索引均可从条目重建。索引只保存稳定 ID、日期、排序值和渲染所需的最小摘要，不复制完整正文。

### 9.2 分页与日期分组

- “全部”和普通分类每页最多 20 条。
- 页面按发布日期倒序，同一天按 `daily_rank` 排序。
- 内容按日期标题分组，不显示时分。
- 使用明确的“更早内容/下一页”分页，不使用无限滚动。
- 日期归档显示指定日期的全部精选。
- 旧详情页永久保持稳定 URL。

二十条是首版阅读窗口和静态生成规模的明确边界，不是精选库上限。

### 9.3 机会生命周期

“机会”分类隐藏“当下重点”，默认先展示仍可行动的内容：

1. `dated` 且截止日期不早于构建日期：按截止日期升序，越临近越靠前。
2. `ongoing`：按发布日期倒序。
3. `unspecified`：标注“未注明截止日期”，按发布日期倒序。

`dated` 且已经过期的内容进入“已截止”分页索引，不在默认机会页出现。`unspecified` 不被系统猜测为过期；它会随分页自然下沉。所有机会详情页永久保留。

## 10. 页面信息架构

### 10.1 产品身份

页面不再把“海南日报精读”作为不可替换的产品身份。产品名应为可配置字符串；首版默认文案为“海南信息雷达”，海南日报只在信息卡的信源位置出现。若后续确定正式名称，只改配置和文案，不改数据契约。

首页顶部只显示产品名、当前视图和“更新至 YYYY-MM-DD”，不显示报纸版面数或报纸文章总数。

### 10.2 全部视图

页面顺序：

1. 产品标题与更新日期。
2. `全部、机会、民生、产业、政策、城市、观察` 选项卡。
3. “当下重点”Top 4。
4. 按日期分组的精选信息卡。
5. 分页入口。

点击当下重点或信息卡进入站内详情页。

### 10.3 分类视图

切换到“机会、民生、产业、政策、城市、观察”后，“当下重点”消失：

- 机会页使用第 9.3 节的生命周期视图。
- 其他分类按日期倒序并分页。
- 某分类为空时保留选项卡并显示简短空状态。

选项卡应使用独立、可直接访问的 URL，而不是只在浏览器中隐藏当前页面节点。刷新和分享分类 URL 后必须保持相同视图。

### 10.4 信息卡

每张信息卡只显示：

- 信源
- 信源原始标题
- AI 摘要

信息卡不显示一级分类、原文链接、版面、最终分、置信度、评分理由、独立的“为什么重要”或事实列表。整张卡片可点击并进入站内详情页。

### 10.5 详情页

稳定地址使用：

```text
/items/YYYY-MM-DD/<item-id>/
```

页面顺序固定为：

1. 分类与发布日期。
2. 信源原始标题。
3. 顶部“查看原文”文字超链接。
4. 视觉上独立的“AI 摘要”区块。
5. 信源抓取到的完整正文，保留原始段落。
6. 底部“查看原文”按钮。

顶部和底部入口都打开 `original_url`。正文不改写、不补写，不把 AI 生成内容混入原文区域。

## 11. 失败处理与发布原子性

以下任一情况使当前生成失败：

- 适配器缺少信源、标题、正文、原始 URL 或发布日期。
- URL 不是有效 HTTP/HTTPS。
- 模型候选缺失、错序、重复或指纹不一致。
- 摘要为空，分类非法或一条内容出现多个一级分类。
- 评分不是 0–10 整数，评分理由缺失。
- 模型返回或试图覆盖信源、标题、正文、URL。
- 机会生命周期与分类不一致。
- 截止日期格式非法或缺少正文证据。
- 条目、索引、首页和详情页的稳定 ID、日期或排名不一致。

所有条目与索引先在内存中构建并完成契约校验，再写入同目录临时文件。整站渲染到 staging 目录；只有所有页面完成并通过链接检查后才替换公开 `site` 目录。失败时保留上一版有效精选库和站点。

抓取失败或正文为空的内容不进入模型输入、精选池或详情页，但在内部预过滤审计中保留原因。

流水线提供一个面向定时任务的统一入口，接受可选 `YYYY-MM-DD`：显式日期用于回放和验收；省略日期时沿用 `crawler.py` 的最新一期发现逻辑。该入口必须从抓取开始，依次完成适配、模型交接、精选、条目写入、索引重建、静态渲染和验证，不得假设目标日期的原始 JSON 已经存在。任一步骤失败均返回非零并保留上一版有效状态。

## 12. 安全与渲染约束

- 标题、摘要、正文、信源和日期全部做 HTML 转义。
- 原文链接只接受 HTTP/HTTPS，并使用安全的外链属性。
- 正文按纯文本段落渲染，不执行信源 HTML 或脚本。
- URL、正文和标题始终来自适配器；模型字段不能影响链接目标。
- 索引页不得内嵌完整正文，控制页面体积。

## 13. 测试与验收

### 13.1 数据与适配器

- 海南日报真实样本的每篇有效文章都转换为统一信息块。
- 标题、正文和 URL 与爬虫原始数据完全一致。
- 新的假信源适配器可以在不修改编辑核心的情况下生成相同契约。
- 稳定 ID 在重复运行中不变化。

### 13.2 模型契约

- 模型只能生成摘要、唯一分类、评分、理由和机会证据。
- 覆盖来源字段、非法分类、多分类、错序、漏项和非法截止日期都会失败。
- 分类定义中的代表性文章都有边界测试，反腐归“观察”，娱乐归“民生”。

### 13.3 精选与当下重点

- 所有超过资格线的条目均入选，超过 8 条时不截断。
- 双资格线、稳定排序和少于四条时不补齐均有测试。
- 当下重点只使用最近三个有内容的日期，按确定性焦点分取前四。
- 新增第四个日期后，最旧日期自动退出焦点候选池。
- 评分核心不读取版面号或版面名。

### 13.4 增长、分页与机会

- 21 条内容生成两页索引，第一页恰好 20 条。
- 同日条目按排名、跨日条目按日期稳定排序。
- 普通分类页隐藏当下重点。
- 未截止机会按截止日期升序，过期机会进入已截止归档。
- `ongoing` 和 `unspecified` 按各自规则展示，系统不猜测截止日期。
- 多年、数千条构造数据只生成当前页面所需摘要，不把完整库加载到单一页面。

### 13.5 页面与链接

- 首页不出现报纸文章总数、版面数、版面号、最终分或置信度。
- 信息卡只显示信源、标题、摘要，并链接正确的站内详情页。
- 分类 URL 可直接打开、刷新和分享。
- 详情页按固定顺序显示分类与日期、标题、顶部原文链接、AI 摘要、完整正文和底部原文按钮。
- 所有站内链接和外链都经过构建后检查。
- HTML 转义、空分类、无精选内容、移动端和分页布局均有测试。

### 13.6 真实样本验收

使用 2026-07-08 原始数据完成 prepare、模型输出校验、finalize、索引构建、整站渲染和视觉检查。验收时确认：

- 公积金、科技见习等高行动价值内容进入精选。
- 科技见习归“机会”。
- 首页标题来自爬虫原始标题。
- 当下重点最多四条。
- 所有合格精选均在“全部”视图中可达。
- 每个信息卡和热点都有可访问的详情页。

### 13.7 新日期增量与定时任务验收

在 2026-07-08 已存在的状态上，使用定时任务将调用的统一入口显式处理下一期：

```bash
bash scripts/run_radar_pipeline.sh 2026-07-09
```

该统一入口内部必须实际执行 `python3 ../hndaily-skill/crawler.py 2026-07-09`，随后继续完成该日期的模型交接、精选、索引和渲染。验收不能只复用或手工修改 2026-07-08 fixture。检查项：

- 爬虫成功获得 2026-07-09，输出日期、版面数和文章数与原始 JSON 一致。
- 新数据无需人工复制字段或编辑中间 JSON 即可进入统一信息块和模型输入。
- 模型输出通过严格契约校验，所有合格条目写入 `content/items/2026-07-09/`。
- 2026-07-08 条目、日期归档和详情页仍然存在且内容未被覆盖。
- “全部”第一页出现 2026-07-09 日期组，并可继续访问 2026-07-08。
- “当下重点”按最近内容日重新计算，索引中的条目均可打开对应详情页。
- 分类索引和机会有效/已截止索引包含新日期带来的正确变化。
- 站点完整渲染、内部链接检查和数据契约验证全部通过。
- 同一日期重复运行保持幂等，不产生重复条目、重复索引记录或不稳定 URL。
- 整个入口退出码为 0；任一模拟失败场景退出非零且公开站点仍保持上一版。

完成这项验收，才可以认为未来按“抓取最新一期”方式运行的定时任务具备可用的端到端基础。

## 14. 实施边界与迁移顺序

本设计可以在一个实施计划中完成，按以下边界拆分：

1. 建立统一信息块、海南日报适配器和新版模型契约。
2. 修改信源无关评分、取消数量上限和旧聚类依赖。
3. 建立按日期条目存储、分类索引、机会生命周期和分页。
4. 重建首页、分类视图和详情页。
5. 增加 staging 整站构建、链接检查和原子替换。
6. 迁移 2026-07-08 数据、更新操作文档并完成真实样本验收。

现有 `content/daily` 契约在迁移期只作为兼容输入；新页面以 `content/items` 和可重建索引为事实来源。周报暂时保留为次级入口，后续再明确它如何从精选库生成。
