# HNHOT 首次文章语义提示词 V4.3

## 任务

逐篇读取《海南日报》报道的标题和正文，严格按照 `schema.json` 输出当前文章的结构化 JSON。

仅使用每条输入中的标题、正文、`published_date`、`location_candidates` 和顶层 `topic_categories`。不得使用外部知识补全事实。主体、事件、规划和细主题从原文自然发现；主题大类只能从输入目录中选择。

只返回严格 JSON，不输出 Markdown 或解释。保持条目顺序和 `candidate_id` 不变，原样返回顶层版本与指纹。所有 `evidence` 必须从当前标题或正文连续逐字复制，不得概括、拼接、删词、调整语序或替换标点。没有可靠内容的数组返回 `[]`；可选字段没有可靠值时省略，不用 `null` 或空字符串占位。

## `ai_summary`

写一句面向普通读者的简洁事实摘要，直接说明最值得知道的问题、变化、行动或结果。保留真正影响理解的地点、日期、数字、政策影响、项目进展或下一步安排；删除会议流程、礼仪动作、口号、空泛评价、重复背景和无关行政过程。正文缺失时返回 `null`，不得只凭标题推测。

## `scope` 与 `scope_evidence`

判断报道主要提供哪种观察视角，不只看发生地：

- `hainan`：海南内部的生活、民生治理、产业、科技、规划建设、基础设施、生态文化或公共服务；
- `domestic`：海南与中国其他省区市发生直接联系，且跨省联系是核心信息；
- `mixed`：海南与境外直接发生联系，或核心价值明确面向开放和国际交流；
- `national`：全国性或中国其他地区信息，没有海南主体直接参与；
- `foreign`：主要事件和行动主体在中国以外，没有海南主体直接参与。

`scope_evidence` 复制一段最能支持分类的连续原文。

## `subjects`

完整提取具有明确动作、结果或影响的人物、企业和机构。不要只选一个代表主体，也不要因为主体可能只出现一次就省略。记者、普通引语提供者和纯礼节性陪同者通常不提取。

- `name`：原文规范名称，不含头衔；
- `type`：`person`、`company` 或 `organization`；
- `role`：原文在本篇明确给出的身份或作用，没有可靠内容时省略；
- `aliases`：只有原文用“以下简称”“简称”“下称”“又称”或“俗称”明确声明时输出，每个别名保留自己的连续证据；没有别名时省略；
- `activities`：该主体在本篇中明确实施、参与、宣布、推动、完成或取得的动作与结果。每个主体至少一项；一项只表达一个可独立理解的动作。只有静态提及、没有明确动作或结果的对象，不进入 `subjects`。

每个 `activities` 项：

- `occurred_on`：动作发生时间，使用 `YYYY-MM-DD`、`YYYY-MM`、`YYYY` 或 `起始日期/结束日期`；无法可靠规范时省略；
- `place`：原文明示的最具体动作发生地，如“海口”“上海”“乐东黎族自治县”；不限于海南，也不是行政区 ID；无法可靠判断时省略；
- `headline`：概括该主体做了什么，必须包含主体名称，通常为 12—32 个汉字；不写“文章报道”“有关工作”等空话；
- `detail_kind`：补充信息类型，只能是 `goal`（明确目标）、`result`（已经产生的结果）或 `object`（动作对象）；
- `detail`：与 `detail_kind` 对应的具体内容。不得把目标写成结果；没有补充信息时与 `detail_kind` 一并省略；
- `evidence`：同时支持主体、时间、地点、动作和补充信息的连续原文。证据不能支持全部表述时，缩小表述范围。

同一主体在一篇文章中只输出一次，可以包含多条不同活动。多个主体共同完成一件事时，分别按各自主体视角记录。项目、道路、园区、产品和规划文件不能因为是重要名词就作为主体，除非原文确实把它写成实际行动主体。

## `location_mentions`

只能选择输入 `location_candidates` 中已有的 `location_id`。同一地点只输出最具体行政区；多个互不包含的市县可以同时输出。每项的 `evidence` 必须直接支持报道归入该行政区。没有可靠候选时返回 `[]`，不得创造地点 ID。

## `topics`

- `primary.category_id`：必须且只能从顶层 `topic_categories` 选择一个最能概括全文的大类，并遵守该类的 `definition` 与 `boundary`；
- `primary.evidence`：支持大类判断的连续原文；
- `secondary`：0—3 个模型自由提取的具体主题，每项只含 `name` 和连续原文 `evidence`。

`secondary[].name` 使用普通读者能理解、可跨文章复用的具体名词短语，通常为 2—10 个汉字，例如“服务业提质增效”“青年创业”“义务教育招生”。不要重复大类名称，不写“海南新闻”“发展”“工作”“综合”等低信息词。相同语义优先使用常见、简短、稳定的称呼，避免制造同义词。

## `events`

只提取原文明示、可以独立命名和跨报道聚合的活动、会议、赛事或突发事件。人物、企业和机构的一般行动已经进入 `subjects[].activities`，不得再包装成事件。

- `name`：本届或本次事件的完整名称；
- `event_type`：
  - `recurring_edition`：带年份、届次或周期性品牌的本届活动，如“2026世界人工智能大会”；
  - `named_event`：有明确专名，但原文不能证明属于周期系列的一次性会议、论坛、展览、赛事或公共活动；
  - `incident`：事故、灾害、公共安全或其他突发事件；
- `series_name`：仅在原文能证明属于周期性品牌或届次活动时，填写去掉年份、届次后的系列名；否则省略；
- `occurred_on`：使用与主体活动相同的时间格式，无法可靠规范时省略；
- `evidence`：支持事件名称和时间的连续原文。

例如，“海南省交通运输厅发布规划”是主体动作，不是事件；“海南省推进服务业（第三产业）高质量发展领导小组第七次例会”是有届次的命名会议，可以作为 `recurring_edition`。

## `plans`

只提取标题或正文中用书名号 `《》` 明确写出的规划、纲要、方案、行动计划或其他具有规划安排性质的政策文件。文学作品、影视作品、普通书籍、法律法规和仅被引用的新闻标题不属于规划。

- `name`：保留书名号的完整文件名；
- `mention_type`：
  - `proposed`：提出编制、拟定或形成草案；
  - `reviewed`：正在审议、评审或征求意见；
  - `approved`：已经审议通过或获批，但原文未表明正式发布；
  - `released`：已经正式发布、印发或公布；
  - `implemented`：已经开始实施、施行或执行；
  - `progress`：报道规划实施后的具体进展或成效；
  - `mentioned`：只提到文件，无法确认上述关系；
- `evidence`：同时支持文件名和 `mention_type` 的连续原文。

## `reader_leads`

把自己当成每天认真读完报纸、愿意及时提醒朋友的人。普通新闻只需摘要和归档；只有报道明确提供读者能采取的动作、需要提前准备或应该避免的风险时，才输出提醒。每篇最多 3 条。

`intent` 只能是：

- `apply`：申请资格、补贴、许可或项目；
- `register`：报名、预约、登记或购票；
- `attend`：参加面向公众开放的活动；
- `submit`：提交意见、建议、材料、投诉或线索；
- `lookup`：查询名单、结果、进度或官方信息；
- `use_service`：使用明确开放的公共服务或设施；
- `prepare`：因天气、停水停电、道路调整或服务中断等提前准备；
- `avoid`：原文明示的安全、诈骗、消费或公共风险，需要避免某种行为。

每项包含：

- `headline`：像朋友提醒一样直说“有什么事”，通常为 8—24 个汉字；
- `audience`：原文明示的适用人群，没有时省略；
- `action`：读者具体能做或需要做什么，不得补写原文没有的步骤；
- `window`：原文有办理或影响时间时输出。`text` 逐字保留原话；只有能可靠规范时才填写 `start`、`end`；
- `channel`：原文明示的平台、地点、热线或办理渠道，没有时省略；
- `evidence`：同时支持人群、动作、时间和渠道的连续原文。

`window.start` 和 `window.end` 使用带 `+08:00` 的 ISO 8601 日期时间；只有日期时使用 `YYYY-MM-DD`。模糊表述如“暑假期间”只保留 `text`。是否生成提醒以报道发布时为判断基准：报道发布时仍可执行的事项应保留，即使以后重新处理时已经过期。报道发布时已经结束且没有后续入口、只面向内部机关执行、普通评论、人物特写和成果报道返回 `[]`。
